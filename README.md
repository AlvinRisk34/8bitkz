# 📋 ELNIT — Умная приоритизация обращений граждан

> AI-решение для отборочного этапа **[GovTech Camp]**
>
> **ELNIT** — Enhanced Local Network for Intelligent Triage

Модель приоритизирует обращения граждан в акимат/ЦОН по реальной срочности
(а не по порядку поступления) и объясняет оператору, почему обращение требует
немедленного внимания.

---

## 🎯 Проблема

Обращения в акимат/ЦОН обрабатываются в порядке поступления. Из-за этого
критичные жалобы — прорыв трубы с горячей водой, оголённый провод, семья без
средств к существованию — могут неделями ждать своей очереди среди рутинных
(«покрасьте лавочку», «замените лампочку»), пока оператор физически не дойдёт
до них по списку.

**Последствия:** задержка реакции на реальные угрозы безопасности, здоровью и
жизни граждан; недоверие к госорганам.

## 💡 Решение

Модель анализирует **текст обращения + структурные признаки** (категория, район,
количество повторных обращений по похожей проблеме) и присваивает
**приоритет-скор (0–100)** — вероятность того, что обращение требует
немедленного внимания.

**Кто пользователь:** оператор ЦОН / аналитик акимата, разбирающий входящие
обращения.

**Измеримый результат:** доля срочных обращений, попавших в топ очереди по
приоритету (recall в топ-N) — насколько раньше критичные случаи будут замечены
по сравнению с обработкой по порядку поступления.

## 🖥 Дашборд

Интерактивный Streamlit-дашборд с пятью разделами:

| Раздел | Описание |
|--------|----------|
| **📊 Обзор** | KPI-полоса (всего / срочных / средних / на проверку), распределение скоров по тиерам, топ-5 срочных, нагрузка по месяцам |
| **📋 Очередь обращений** | Фильтры (категория, район, тиер, приоритет, поиск по тексту), карточки с explainability-барами, пагинация |
| **📈 Аналитика** | Обращения по категориям и районам, box-plot скоров, тепловая карта район × категория, повторные обращения vs приоритет |
| **➕ Новое обращение** | Ввод текста → модель считает скор на лету, показывает тиер и explainability-бары — демонстрация работы «вживую» |
| **🧠 О модели** | Описание подхода, ROC-AUC/PR-AUC, ROC/PR-кривые, матрица ошибок, топ-20 важных признаков, ограничения |

### Explainability

Для каждого обращения дашборд показывает **горизонтальные бары вклада**
топ-факторов — какие слова и признаки повысили приоритет, и на сколько.
Например: `прорвало` +0.34, `repeat_count` +0.28, `кипяток` +0.19.

### Роль человека

Модель **не закрывает и не отклоняет обращения**. Она только:
- переупорядочивает очередь по приоритету;
- объясняет, почему обращение попало наверх;
- оставляет финальное решение за оператором.

---

## 📊 AI/ML-компонент

- **Задача:** бинарная классификация / скоринг (обращение срочное или нет)
  → логистическая регрессия на TF-IDF + структурных признаках.
- **Признаки:**
  - текст обращения → TF-IDF (уни- и биграммы, до 2000 признаков);
  - категория обращения (one-hot);
  - количество повторных обращений по похожей проблеме;
  - месяц / день недели.
- **Почему логистическая регрессия, а не бустинг:** коэффициенты линейно
  интерпретируемы — для каждого обращения можно точно показать вклад каждого
  слова/признака (вклад = коэффициент × значение признака), без SHAP и
  дополнительных инструментов.
- **Метрики (синтетика, test split 25%):** ROC-AUC = **0.850**, PR-AUC = **0.712**.

## 🗂 Данные

### Синтетические данные (текущая версия)

1500 обращений, сгенерированных в `src/generate_synthetic_data.py`.

**Логика генерации:** для 5 категорий (ЖКХ, Дороги, Безопасность,
Благоустройство, Соцвопросы) заданы шаблоны текста трёх уровней срочности
(низкая/средняя/высокая). К шаблонам подставляются случайные детали (улица,
район) для разнообразия. Истинная срочность зависит от уровня шаблона +
repeat_count + случайный шум — модель учится находить закономерности в тексте,
а не запоминать шаблон дословно.

| Колонка | Описание |
|---------|----------|
| `appeal_id` | Уникальный идентификатор |
| `date` | Дата поступления |
| `district` | Район Алматы (6 районов) |
| `category` | Категория обращения (5 категорий) |
| `text` | Текст обращения |
| `repeat_count` | Кол-во повторных обращений по похожей проблеме |
| `is_urgent` | Целевая переменная (1 = срочное, 0 = не срочное) |

### Реальные данные (следующий шаг)

Наиболее перспективный источник — **Open Almaty** (open-almaty.kz),
официальная общественная приёмная акимата Алматы (~2 млн обращений
с момента запуска). План:
1. Запросить анонимизированную выгрузку через контакт-центр (1308) или
   фронт-офис (пр. Абая, 90).
2. Если явной разметки срочности нет — создать прокси-метрику
   (например, время до первой реакции оператора).
3. Разметить вручную небольшую выборку (100–300 обращений) для валидации.

Альтернативные источники: [data.egov.kz](https://data.egov.kz),
[eOtinish](https://eotinish.kz/ru).

**Как подключить реальные данные:** положить CSV-файл в `data/raw/appeals.csv`
с колонками `date, district, category, text, repeat_count, is_urgent`
(последняя — целевая переменная; если разметки нет, создать прокси).

## ⚡ Быстрый старт

```bash
# 1. Клонировать и установить зависимости
git clone <repo-url>
cd 8bitkz
pip install -r requirements.txt

# 2. Сгенерировать данные (или положить реальные в data/raw/appeals.csv)
python src/generate_synthetic_data.py

# 3. Обучить модель
python src/train_model.py

# 4. Запустить дашборд
streamlit run app/dashboard.py
```

Дашборд откроется на **http://localhost:8501**.

## ⚠️ Ограничения

- Данные синтетические — на реальных обращениях модель нужно переобучить
  и скорректировать целевую переменную.
- Модель не учитывает вложения (фото/видео) — только текст.
- TF-IDF не понимает синонимы и опечатки так же хорошо, как современные
  языковые модели — при масштабировании заменить на embeddings.
